I build intelligent systems.

I ship production LLM systems — from a Springer-published model that reads chest X-rays well enough for a radiologist to take seriously, to document pipelines that run themselves. Before any of that, I wrote real-time control code for jet engines at DRDO, where a millisecond of lag isn't a bug — it's a flameout.

That job rewired how I build software. Turbine control taught me that the interesting part of a system isn't what it does when everything works. It's what it refuses to do when something goes wrong. You'll find that idea sitting at the centre of this project: the agent isn't clever because it can edit your repo. It's useful because there's a gate that stops it.

Most of what I build lives in the same territory — AI that has to survive contact with real inputs. Plenty of it looks like magic in a notebook and falls apart in production, which is the actual problem worth solving.

This is one project. There are eighteen more — and a few jet engines — over at charanreddy.dev.

What I learned building this

Five things I didn't know in March

Four technical, one about me. All of them cost something to learn.

  1. 01 · Prompts vs. gates A constraint in a prompt is a request, not a rule

    I started by writing "Phase 1 is strictly read-only" in the system prompt and assuming that settled it. It doesn't. The fix was moving the rule into a canUseTool callback that runs before the tool does. Same sentence, completely different guarantee.

  2. 02 · Model output Injecting a fact into a prompt doesn't mean the model uses it

    The first real run stamped the report 2025-01-21 — a date from the model's training prior — with the correct date sitting right there in its own prompt. Now the programme rewrites that line itself. Anything verifiable in code gets verified in code.

  3. 03 · Agent design An agent with no sense of budget will explore until it dies

    At a 30-turn cap, my first run read both repos thoroughly, wrote nothing, and hit the limit. Zero output. Telling the agent its budget and instructing it to spend at most half on exploring fixed it. Autonomy needs a clock.

  4. 04 · Product The valuable output was the "no"

    I expected the recommendations to be the product. In the verified run, ten of sixteen findings were Skips — and those were the ones that saved real time. Anyone can generate suggestions. Being told which ones don't matter is the scarce thing.

  5. 05 · Human I trusted a demo I hadn't watched fail

    My instinct was to ship after the first clean run. Both real bugs in this project surfaced by deliberately running it badly — a starved turn budget, a report I actually read to the end. Confidence in agent code should come from watching it break, not from watching it work once.
